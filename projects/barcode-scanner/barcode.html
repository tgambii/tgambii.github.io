Overview

The problem it solved

Tracking which parts are pulled from the hardware lab for each build required manually cross-referencing a physical parts list, looking up SAP material numbers, and entering quantities into a spreadsheet by hand. It was slow, error-prone, and entirely dependent on an engineer knowing the SAP part numbering system by memory.

The goal was to replace that process with a barcode scanner — scan the label on a bin, the app resolves the SAP part number instantly, shows a photo of the item for visual confirmation, takes a quantity, and at the end of the build exports a clean Excel report ready for SAP entry.

Interface

Application GUI

Built with Python's tkinter library. The left panel handles scanning and confirmation; the right panel shows a running log of all parts scanned in the current build session.

Technical breakdown

How it works

Parts at Curtiss-Wright follow a consistent naming convention: TCG-HL-{PREFIX}-{NUMBER}. The barcode generation algorithm converts the letter prefix into a numeric sequence, pads the result, and calculates a valid EAN-13 check digit — making the output scannable by any standard USB barcode reader.

# EAN-13 generation — convert SAP P/N to a scannable barcode def generate_barcode(entry, letter_map): prefix, number = entry.split("-")[2], entry.split("-")[3] barcode = "0000" + replace_letters(prefix, letter_map) + number digits = [int(d) for d in barcode] total_sum = sum(digits[i] if i % 2 == 0 else digits[i] * 3 for i in range(12)) check = str((10 - (total_sum % 10)) % 10) return barcode + check

Development process

How it was built

Step 01
Command-line prototype
Started as a pure terminal application. Input a barcode number, get back a SAP P/N and prompt for quantity. Validated the EAN-13 algorithm against real part numbers before building any UI.
Step 02
EAN-13 barcode map
Built the letter-to-digit encoding system and check digit algorithm to convert the company's proprietary part numbering format into valid EAN-13 barcodes that any off-the-shelf USB scanner can read.
Step 03
SAP API integration
Designed the API client to pull live Material Master data from SAP ERP using OData REST. Implemented secure credential handling via environment variables. Hit a blocker — IT approval for the endpoint was not granted.
Step 04
Desktop GUI
Built the full tkinter interface to replace the command-line flow — part image display, quantity confirmation, live scanned-items table, and one-click Excel export named after the build number.
Step 05
Manual fallback & packaging
With the API blocked, added a manual parts list fallback so the tool is still fully usable today. Explored PyInstaller packaging for distribution to Windows lab machines that don't have Python installed.
⛔
Deployment blocker The SAP BASIS team did not provide an approved OData API endpoint for the Material Master service. The application is production-ready and the API client is fully implemented — it requires only a valid endpoint URL and credentials to go live. The project remains deployed in manual-list mode pending IT approval.
