Barcode Scanner with SAP Integration

A Python desktop application built to streamline parts tracking in the Curtiss-Wright hardware lab — scanning physical barcodes, outputting SAP part numbers, displaying part images, and exporting build reports to Excel.

Overview

The Problem.

Tracking which parts are pulled from the hardware lab for each build required manually cross-referencing a physical parts list, looking up SAP material numbers, and entering quantities into a spreadsheet by hand.It was slow, error-prone, and entirely dependent on an assembler knowing the SAP part numbering system by memory. The goal was to replace that process with a barcode scanner — scan the label on a bin, get the SAP part number instantly, see a photo of the part for visual confirmation, enter a quantity, and export a clean report at end of a build.

Barcode resolution
USB scanner input matched against a built EAN-13 lookup table to return the SAP part number instantly.
Part image display
On match, the app loads that specific SAP_PN.png from a local folder for visual confirmation before quantity is committed.
Excel export
Quantities are written to a formatted .xlsx file named after the build number, ready for SAP entry.
SAP API hook
Architecture supports live Material Master fetching via SAP OData REST API with secure credential management.

Technical Breakdown

How it Works.

Architecture

The project is structured as a single-package Python application with four distinct layers:

Barcode Scanner/
├── main.py # tkinter GUI — entry point
├── requirements.txt # packages needed to install
├── .env # SAP_USERNAME / SAP_PASSWORD
├── core/
├── __init__.py # gives access to SAP_barcode_map, barcode_readout
├── barcode_map.py # EAN-13 generation + lookup table
├── sap_api.py # SAP Material Master API client
└── replace_letters.py # letter → digit encoder
├── Part Number Images/ # SAP_PN.png files
├── TCG-HL-KM-1008.png # .pngs for visual confirmation embedded in GUI
└── etc.
└── exports/ # auto-created on first export, .xlsx output
└── 2025-3.xlsx # .pngs for visual confirmation embedded in GUI

Module Breakdown

The application is split into four modules, each owning a single layer of responsibility. Data flows top-to-bottom: SAP API → barcode generation → GUI controller → Excel export.


1 — sap_api.py — SAP Integration Layer

  • Loads credentials from a .env file using python-dotenv, keeping secrets out of source control.
  • Issues an authenticated HTTP GET to a SAP REST endpoint using HTTPBasicAuth.
  • Parses a JSON array response and extracts MaterialNumber fields, returning a flat list[str].
  • Gracefully degrades: on any network failure or non-200 response, returns None, allowing the app to fall back to a hardcoded MANUAL_PARTS list. This makes the tool functional in air-gapped or offline lab environments.


2 — replace_letters.py — Encoding Utility

  • A pure, stateless function that maps alphabetic characters in a part number prefix to their positional equivalents (A→01, B→02, … Z→26).
  • This is the key transformation that converts human-readable SAP part number segments into an all-numeric string, a prerequisite for EAN-13 barcode generation.


3 — barcode_map.py — EAN-13 Generation & Data Model

This is the computational core of the application. It executes at import time, building all lookup structures the GUI depends on. It calls get_material_numbers() to fetch live SAP data (or uses fallback MANUAL_PARTS list).

Parts at Curtiss-Wright follow a consistent naming convention: TCG-HL-{PREFIX}-{NUMBER}. The barcode generation algorithm converts the letter prefix to a numeric sequence, concatenates it with the 4 digit number, pads the result to 12 digits, and calculates a valid EAN-13 check digit — making the output scannable by any standard USB barcode reader. The function generates two representations: the full 13-digit EAN-13 barcode, and a "readout" string (the format a USB HID barcode scanner actually transmits to the host, which drops a leading zero).

# EAN-13 generation — convert SAP P/N to a scannable barcode
def generate_barcode(entry, letter_map):
prefix, number = entry.split("-")[2], entry.split("-")[3]
barcode = "0000" + replace_letters(prefix, letter_map) + number
digits = [int(d) for d in barcode]
total_sum = sum(digits[i] if i % 2 == 0 else digits[i] * 3 for i in range(12))
check = str((10 - (total_sum % 10)) % 10)
barcode += check_digit
readout = "000" + replace_letters(prefix, letter_map) + number + check_digit
return barcode,readout


4 — main.py — GUI Application Controller

  • BarcodeScannerApp subclasses tk.Tk and owns the full application lifecycle.
  • Uses tk.Label widgets styled as buttons instead of tk.Button — this is an intentional workaround for macOS's Aqua theme, which overrides background/foreground colors on native button widgets, breaking the custom color scheme.
  • Scan flow: Entry widget captures barcode input (keyboard or USB HID scanner via Enter key binding) → O(1) dict lookup against SAP_barcode_map → updates SAP P/N label, loads part image from Part Number Images/.png, sets status.
  • Image rendering: PIL/Pillow is used to load, convert to RGBA, thumbnail-scale, and composite each part image onto a fixed-size canvas before converting to ImageTk.PhotoImage. The canvas approach preserves aspect ratio and keeps the UI consistent regardless of source image dimensions.
  • Session state: self.excel: dict[str, int] accumulates SAP P/N → quantity, supporting additive scanning (scanning the same part twice increments its count).
  • Export: Writes the session dict to a named Excel file (exports/{BuildNumber}.xlsx) using openpyxl, with cells explicitly styled as text format ("@") to prevent Excel from cutting out the leading-zero barcodes. Then calls print_map() to refresh the barcode reference sheet.
  • Cross-platform theming: Detects platform.system() == "Darwin" to select the aqua ttk theme on macOS (required to render Treeview column headers correctly), falling back to clam on Windows.
